## 프로젝트 개요

clubfair-2026은 동아리 박람회 참여자를 대상으로 성향 테스트, 동아리 매칭, 이벤트 참여 흐름을 제공한 실제 운영 서비스입니다. 2025년에는 로컬 노트북 한 대로 서비스를 운영했지만, 단일 기기 의존, 데이터 휘발, 실시간 모니터링 불가라는 한계가 있었습니다.

2026년에는 이 경험을 바탕으로 Supabase와 Vercel 기반의 클라우드 SaaS 아키텍처로 전환하는 것을 목표로 삼았습니다. 프론트엔드 화면 구현뿐 아니라, 서비스 운영에 필요한 데이터 흐름과 Supabase 기반 운영 구조를 함께 고민한 프로젝트입니다.

## 문제 정의

박람회 현장 서비스는 짧은 시간에 많은 사용자가 동시에 접속합니다. 단순히 화면이 잘 보이는 것만으로는 충분하지 않고, 이벤트 참여와 재화 차감처럼 데이터 무결성이 필요한 흐름이 안전하게 처리되어야 합니다.

특히 다수의 사용자가 이벤트에 동시에 응모할 경우 재화가 중복 차감되거나 당첨자가 초과 선정될 수 있었습니다. 처음에는 프론트엔드 레벨에서 중복 클릭이나 중복 요청을 막는 방식으로 접근했지만, 클라이언트만으로는 근본적인 동시성 문제를 해결할 수 없다는 것을 확인했습니다.

## 내가 맡은 역할

Frontend / Cloud Architecture / Supabase 기반 운영 구조 기여 역할로 참여했습니다. 팀 프로젝트이므로 전체 서비스의 모든 구현을 혼자 담당한 것은 아니며, 사용자 화면과 상태 관리, 클라우드 전환 방향, Supabase를 활용한 운영 구조와 데이터 무결성 흐름에 기여한 범위를 중심으로 정리합니다.

## 핵심 구현

- React와 TypeScript 기반 사용자 화면 구현
- TanStack Query로 서버 상태, Zustand로 클라이언트 상태를 분리
- Supabase와 PostgreSQL을 활용한 운영 데이터 흐름 설계에 기여
- 이벤트 응모 흐름에서 PostgreSQL `FOR UPDATE` 기반 Row Lock 트랜잭션을 활용해 데이터 무결성 보장
- RLS 정책과 `SECURITY DEFINER` 함수를 통해 민감한 비즈니스 로직을 데이터베이스 계층에서 보호
- Vercel 배포와 Supabase 기반 운영 구조를 통해 로컬 노트북 의존 운영에서 클라우드 기반 운영으로 전환

## 기술적 판단

동시성 문제는 프론트엔드에서 버튼을 비활성화하거나 요청을 한 번만 보내도록 처리하는 것으로 완전히 해결할 수 없었습니다. 사용자는 여러 탭, 느린 네트워크, 재요청 상황을 만들 수 있고, 클라이언트 상태는 신뢰할 수 있는 최종 방어선이 아니기 때문입니다.

그래서 이벤트 응모와 재화 차감처럼 정합성이 중요한 로직은 데이터베이스 계층에서 처리하는 방향이 더 적절하다고 판단했습니다. PostgreSQL Row Lock 트랜잭션을 사용하면 같은 행을 동시에 수정하려는 요청을 순서대로 처리할 수 있고, RLS와 `SECURITY DEFINER` 함수는 클라이언트가 직접 민감한 로직을 우회하지 못하도록 보호하는 데 도움이 되었습니다.

## 어려웠던 점과 해결

가장 어려웠던 점은 프론트엔드 개발자 관점에서 익숙한 UI 상태 문제와 데이터베이스 트랜잭션 문제를 분리해 바라보는 것이었습니다. 처음에는 사용자 입력을 막는 방식으로 중복 참여를 줄이려 했지만, 실제 운영 서비스에서는 서버와 데이터베이스에서 보장해야 하는 규칙이 따로 있다는 것을 배웠습니다.

이를 해결하기 위해 이벤트 응모 흐름을 클라이언트 상태, 서버 요청, DB 트랜잭션, 보안 정책으로 나누어 다시 정리했습니다. 프론트엔드는 사용자가 현재 상태를 명확히 이해하도록 돕고, 데이터 무결성은 데이터베이스 계층에서 보장하는 구조로 역할을 분리했습니다.
