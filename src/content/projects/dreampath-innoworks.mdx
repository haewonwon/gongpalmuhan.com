## 프로젝트 개요

DreamPath / InnoWorks는 AI와 대화하며 시각화 결과를 확인할 수 있는 프로그램입니다. 기존 Django 풀스택 구조를 Django 백엔드와 React 프론트엔드로 분리하는 과정에 참여했고, 분리된 프론트엔드가 서비스 상태와 AI 응답을 안정적으로 표현할 수 있도록 화면 구조와 상태 관리 기준을 정리했습니다.

## 문제 정의

기존 구조에서는 서버 렌더링과 프론트엔드 화면 로직이 한 프로젝트 안에 묶여 있어 화면 변경, 상태 관리, 다국어 확장, 배포 자동화를 독립적으로 개선하기 어려웠습니다. AI 응답을 보여주는 서비스 특성상 로그인 상태, 진행도, 주차 상태, 프롬프트 상태가 UI와 명확히 연결되어야 했고, 다국어 사용자도 같은 흐름을 이해할 수 있어야 했습니다.

## 내가 맡은 역할

Frontend / Migration / CI/CD 역할로 참여했습니다. 전체 제품과 백엔드 구현을 단독으로 담당한 것은 아니며, 기존 Python 서버 코드를 AI Agent와 함께 구조/기능 단위로 빠르게 파악한 뒤 React 프론트엔드로 분리하는 과정, 화면 상태 관리 기준, 다국어 UI 반영, GitHub Actions 기반 배포 자동화 쪽에 기여했습니다.

## 핵심 구현

- Django 중심 구조를 Django 백엔드와 React 프론트엔드로 분리하는 마이그레이션에 참여
- React 서비스에 맞는 프론트엔드 아키텍처와 상태 관리 구조 정의
- 영어, 일본어, 중국어, 스페인어 다국어 지원 구조를 설계하고 UI에 반영
- GitHub Actions 기반 CI/CD 파이프라인 구축
- 진행도, 주차 상태, 로그인 상태 등 서버 데이터를 화면 상태로 반영
- 상태에 따라 프롬프트가 명확히 달라지는 프롬프팅 구조 설계
- AI 응답 UI를 Markdown이 아닌 HTML 렌더링 방식으로 구성

## 기술적 판단

프론트엔드를 React로 분리하면서 화면 상태와 서버 상태의 책임을 더 명확히 나누는 것을 우선했습니다. AI 응답은 일반 문서처럼 Markdown으로만 다루기보다, 서비스 UI 안에서 의도한 구조와 스타일을 유지해야 했기 때문에 HTML 형태로 렌더링하는 방향을 선택했습니다.

또한 기존 Python 서버 코드를 처음부터 모두 수작업으로 읽기보다, AI Agent를 활용해 구조와 기능 단위를 빠르게 파악한 뒤 사람이 검토하는 방식으로 마이그레이션 속도를 높였습니다. 생성된 이해나 제안은 그대로 신뢰하지 않고 실제 코드 흐름과 맞는지 확인하면서 반영했습니다.

## 어려웠던 점과 해결

가장 어려웠던 점은 기존 서버 중심 구조를 React 화면 구조로 옮기면서도 기능 의미를 잃지 않는 것이었습니다. 특히 AI 대화 흐름은 단순한 텍스트 출력이 아니라 사용자의 상태와 연결되어 있었기 때문에, 상태별 프롬프트와 응답 표시 방식을 먼저 정리한 뒤 UI에 반영했습니다.

다국어 지원 역시 텍스트 치환만으로 끝나지 않았습니다. 언어별 문장 길이와 화면 흐름을 고려해 UI에서 자연스럽게 노출될 수 있도록 구조를 분리했습니다.

## 결과 / 배운 점

프론트엔드와 백엔드를 분리한 뒤 화면 변경과 배포 자동화, 다국어 확장, 상태 기반 UI 개선을 더 독립적으로 다룰 수 있는 기반을 만들었습니다. 내부 구현 세부사항은 공개하지 않지만, 인턴십 프로젝트를 통해 기존 코드베이스를 빠르게 파악하고 서비스 구조를 안전하게 옮기는 경험을 했습니다.

이 프로젝트를 통해 AI 도구는 코드를 대신 작성하게 두는 도구라기보다, 낯선 코드베이스를 빠르게 이해하고 리팩토링 방향을 정리하는 보조 도구로 사용할 때 효과적이라는 점을 배웠습니다.
